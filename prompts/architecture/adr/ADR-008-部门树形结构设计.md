# ADR-008: 部门树形结构设计

## 状态
Accepted

## 背景
用户角色权限管理系统需要支持企业组织架构管理，根据FRD v1.2需求，系统需要支持五级部门层级（公司→一级部门→二级部门→三级部门→四级部门）。需要选择合适的数据结构来存储和查询部门树形关系。

## 决策
采用 **Materialized Path（物化路径）** 模式存储部门树形结构，配合层级字段（level）和路径字段（path）。

## 理由

### 考虑的方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **邻接表 (Adjacency List)** | 简单，易于理解，插入/删除快 | 查询子树需要递归，性能差 | 层级浅（<5层），数据量小 |
| **物化路径 (Materialized Path)** | 查询子树快（LIKE查询），无需递归 | 更新层级复杂，路径长度有限 | 层级固定，查询频繁 |
| **嵌套集 (Nested Sets)** | 查询子树和祖先快 | 插入/删除复杂，容易死锁 | 读多写少，层级变动少 |
| **闭包表 (Closure Table)** | 灵活，支持任意深度 | 空间占用大，查询稍复杂 | 层级深度变化大 |

### 选择 Materialized Path 的原因

1. **查询性能优秀**
   - 查询子树：`WHERE path LIKE '/1/2/5%'`
   - 查询路径：`WHERE id IN (1, 2, 5)`
   - 查询层级：`WHERE level = 3`

2. **符合业务需求**
   - 业务限制最多5级，路径长度可控
   - 部门结构调整相对较少
   - 查询需求远大于更新需求

3. **实现简单**
   - 数据库表设计简单
   - 与JPA集成容易
   - 团队理解成本低

4. **缓存友好**
   - 完整部门树可以缓存为JSON
   - 子树查询结果可以单独缓存

### 具体设计

```sql
CREATE TABLE department (
    id UUID PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    code VARCHAR(50) UNIQUE NOT NULL,  -- 部门编码，如 DEPT-001
    parent_id UUID REFERENCES department(id),
    level INT NOT NULL CHECK (level BETWEEN 1 AND 5),
    path VARCHAR(500) NOT NULL,        -- 格式: /1/2/5/10
    sort_order INT DEFAULT 0,
    manager_id UUID REFERENCES user(id),
    description VARCHAR(500),
    status VARCHAR(20) DEFAULT 'ACTIVE',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 关键索引
CREATE INDEX idx_department_path ON department(path);
CREATE INDEX idx_department_parent ON department(parent_id);
CREATE INDEX idx_department_level ON department(level);
```

### 路径格式说明
- 根部门：`/1`
- 一级部门：`/1/10`
- 二级部门：`/1/10/100`
- 三级部门：`/1/10/100/1000`
- 四级部门：`/1/10/100/1000/10000`

## 后果

### 正面影响
1. **查询性能好**：子树查询只需一次LIKE操作
2. **实现简单**：代码逻辑清晰，易于维护
3. **缓存友好**：适合Redis缓存部门树结构
4. **扩展性好**：支持数据权限的部门范围过滤

### 负面影响
1. **更新复杂**：调整部门层级需要更新所有子部门的路径
2. **路径长度限制**：VARCHAR(500)可能限制深度（但业务限制5级足够）
3. **并发更新风险**：批量更新路径时需要事务控制

### 缓解措施
1. **更新优化**：使用递归CTE或应用层逻辑批量更新路径
2. **事务控制**：使用乐观锁或悲观锁防止并发更新冲突
3. **缓存失效**：部门结构调整后清除相关缓存
4. **监控告警**：监控路径长度，接近限制时告警

## 替代方案

### 方案A：邻接表 + 递归查询
```sql
-- PostgreSQL递归查询
WITH RECURSIVE department_tree AS (
    SELECT id, name, parent_id, 1 as level
    FROM department
    WHERE parent_id IS NULL
    UNION ALL
    SELECT d.id, d.name, d.parent_id, dt.level + 1
    FROM department d
    INNER JOIN department_tree dt ON d.parent_id = dt.id
)
SELECT * FROM department_tree;
```
**放弃原因**：递归查询性能差，不适合高并发场景。

### 方案B：闭包表
```sql
CREATE TABLE department_closure (
    ancestor_id UUID NOT NULL,
    descendant_id UUID NOT NULL,
    depth INT NOT NULL,
    PRIMARY KEY (ancestor_id, descendant_id)
);
```
**放弃原因**：空间占用大，插入/删除复杂，查询性能不如物化路径。
