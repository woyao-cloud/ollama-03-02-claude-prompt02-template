# ADR-009: 数据权限范围实现

## 状态
Accepted

## 背景
根据FRD v1.2需求，系统需要支持细粒度的数据权限控制。除了菜单、操作、字段权限外，还需要支持数据行级别的权限控制。具体需要实现四种数据权限范围：全部(ALL)、本部门(DEPT)、本人(SELF)、自定义(CUSTOM)。

## 决策
采用 **角色级数据范围配置** + **查询时动态过滤** 的方式实现数据权限控制。

## 架构设计

### 1. 数据模型设计

```sql
-- 角色表增加数据范围字段
ALTER TABLE role ADD COLUMN data_scope VARCHAR(20) NOT NULL DEFAULT 'SELF';
ALTER TABLE role ADD COLUMN data_conditions JSONB;

-- 数据范围枚举
-- ALL: 全部数据
-- DEPT: 本部门及子部门数据
-- SELF: 本人创建的数据
-- CUSTOM: 自定义条件
```

### 2. 权限检查流程

```
用户请求 → JWT认证 → 获取用户角色 → 查询数据范围配置
                                         ↓
                               根据data_scope生成过滤条件
                                         ↓
                               应用到查询语句
                                         ↓
                               返回过滤后的数据
```

### 3. 实现方案选择

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **方案A: 数据库视图** | 为每个角色创建数据库视图 | 性能好，数据库级安全 | 动态创建视图复杂，维护困难 |
| **方案B: 应用层过滤** | 在应用层添加WHERE条件 | 灵活，易于实现和维护 | 性能依赖应用层实现 |
| **方案C: 行级安全** | 使用PostgreSQL行级安全策略 | 数据库级安全，性能好 | 配置复杂，跨数据库不兼容 |
| **方案D: 数据标记** | 每条数据标记可见范围 | 简单直观 | 数据冗余，更新复杂 |

**选择方案B（应用层过滤）的原因**：
1. **灵活性高**：支持复杂的自定义条件
2. **易于维护**：代码控制，无需数据库特殊配置
3. **跨数据库兼容**：不依赖特定数据库特性
4. **团队熟悉**：Spring Data JPA + Specification模式

## 具体实现

### 1. 数据范围枚举

```java
public enum DataScope {
    ALL("全部数据"),
    DEPT("本部门及子部门数据"),
    SELF("本人数据"),
    CUSTOM("自定义范围");

    private final String description;

    DataScope(String description) {
        this.description = description;
    }
}
```

### 2. 数据权限拦截器

```java
@Component
public class DataPermissionInterceptor {

    @Autowired
    private DepartmentService departmentService;

    /**
     * 根据数据范围生成过滤条件
     */
    public <T> Specification<T> createDataFilter(
            UserDetails userDetails,
            DataScope dataScope,
            String customConditions) {

        CustomUserDetails user = (CustomUserDetails) userDetails;

        switch (dataScope) {
            case ALL:
                return null; // 无过滤

            case DEPT:
                return createDepartmentFilter(user.getDepartmentId());

            case SELF:
                return (root, query, cb) ->
                    cb.equal(root.get("createdBy"), user.getId());

            case CUSTOM:
                return parseCustomConditions(customConditions, user);

            default:
                throw new IllegalArgumentException("未知的数据范围: " + dataScope);
        }
    }

    /**
     * 创建部门过滤条件
     */
    private <T> Specification<T> createDepartmentFilter(UUID departmentId) {
        return (root, query, cb) -> {
            // 获取部门子树ID列表
            List<UUID> accessibleDeptIds = departmentService
                .getSubDepartmentIds(departmentId);
            accessibleDeptIds.add(departmentId);

            return root.get("departmentId").in(accessibleDeptIds);
        };
    }

    /**
     * 解析自定义条件
     */
    private <T> Specification<T> parseCustomConditions(
            String conditions, CustomUserDetails user) {
        // 解析JSON格式的自定义条件
        // 例如: {"departmentId": "dept123", "status": "ACTIVE"}
        // 支持变量替换，如 ${userId}, ${departmentId}
        return DataConditionParser.parse(conditions, user);
    }
}
```

### 3. 在Service层应用过滤

```java
@Service
public class UserService {

    @Autowired
    private DataPermissionInterceptor dataPermissionInterceptor;

    @Autowired
    private UserRepository userRepository;

    public Page<UserDTO> listUsers(Pageable pageable, UserDetails userDetails) {
        // 获取用户的数据范围
        DataScope dataScope = getCurrentUserDataScope(userDetails);

        // 创建数据过滤条件
        Specification<User> dataFilter = dataPermissionInterceptor
            .createDataFilter(userDetails, dataScope, null);

        // 应用过滤条件查询
        Page<User> users = userRepository.findAll(dataFilter, pageable);

        return users.map(this::toDTO);
    }

    private DataScope getCurrentUserDataScope(UserDetails userDetails) {
        // 获取用户所有角色，取最宽松的数据范围
        // 规则: ALL > DEPT > CUSTOM > SELF
        CustomUserDetails user = (CustomUserDetails) userDetails;
        return user.getRoles().stream()
            .map(Role::getDataScope)
            .max(Comparator.comparingInt(DataScope::getPriority))
            .orElse(DataScope.SELF);
    }
}
```

### 4. 自定义条件解析器

```java
@Component
public class DataConditionParser {

    public static <T> Specification<T> parse(
            String conditionsJson, CustomUserDetails user) {

        ObjectMapper mapper = new ObjectMapper();
        try {
            JsonNode conditions = mapper.readTree(conditionsJson);
            return parseNode(conditions, user);
        } catch (JsonProcessingException e) {
            throw new DataConditionParseException("条件解析失败", e);
        }
    }

    private static <T> Specification<T> parseNode(
            JsonNode node, CustomUserDetails user) {

        List<Specification<T>> specs = new ArrayList<>();

        node.fields().forEachRemaining(entry -> {
            String field = entry.getKey();
            JsonNode value = entry.getValue();

            // 支持变量替换
            String stringValue = value.asText();
            stringValue = replaceVariables(stringValue, user);

            specs.add((root, query, cb) ->
                cb.equal(root.get(field), stringValue));
        });

        // 组合所有条件（AND关系）
        return Specification.allOf(specs);
    }

    private static String replaceVariables(String value, CustomUserDetails user) {
        return value
            .replace("${userId}", user.getId().toString())
            .replace("${departmentId}", user.getDepartmentId().toString())
            .replace("${userEmail}", user.getEmail());
    }
}
```

## 性能优化

### 1. 部门子树缓存

```java
@Service
public class DepartmentService {

    @Cacheable(value = "departmentSubtree", key = "#departmentId")
    public List<UUID> getSubDepartmentIds(UUID departmentId) {
        String path = departmentRepository.findPathById(departmentId);
        return departmentRepository.findIdsByPathStartingWith(path + "/");
    }
}
```

### 2. 数据权限缓存

```java
@Component
public class DataPermissionCache {

    @Cacheable(value = "userDataScope", key = "#userId")
    public DataScope getUserDataScope(UUID userId) {
        // 查询数据库获取用户的数据范围
        return userRepository.findDataScopeByUserId(userId);
    }
}
```

### 3. 批量查询优化

```java
public class BatchDataFilter {

    /**
     * 批量应用数据权限过滤
     */
    public <T> List<T> filterByDataScope(
            List<T> entities,
            UserDetails userDetails,
            DataScope dataScope) {

        if (dataScope == DataScope.ALL) {
            return entities;
        }

        return entities.stream()
            .filter(entity -> hasDataPermission(entity, userDetails, dataScope))
            .collect(Collectors.toList());
    }
}
```

## 后果

### 正面影响
1. **细粒度控制**：支持四种数据范围，满足企业级需求
2. **灵活性高**：自定义条件支持复杂业务场景
3. **易于扩展**：可以轻松添加新的数据范围类型
4. **性能可控**：通过缓存和优化减少性能影响

### 负面影响
1. **性能开销**：应用层过滤相比数据库级方案有额外开销
2. **实现复杂度**：需要处理各种边界情况和错误处理
3. **缓存一致性**：部门结构调整时需要更新相关缓存

### 缓解措施
1. **性能优化**：使用缓存，优化查询，批量处理
2. **代码质量**：编写单元测试，代码审查，文档完善
3. **监控告警**：监控数据权限查询性能，设置阈值告警

## 替代方案

### 方案A：数据库行级安全（RLS）
```sql
-- PostgreSQL行级安全策略
ALTER TABLE user ENABLE ROW LEVEL SECURITY;
CREATE POLICY user_access_policy ON user
    USING (department_id IN (
        SELECT get_accessible_departments(current_user)
    ));
```
**放弃原因**：配置复杂，跨数据库不兼容，团队不熟悉。

### 方案B：数据标记法
```sql
-- 每条数据标记可见范围
ALTER TABLE user ADD COLUMN visibility VARCHAR(20);
-- 值: PUBLIC, DEPARTMENT, PRIVATE
```
**放弃原因**：数据冗余，更新复杂，不支持动态条件。
