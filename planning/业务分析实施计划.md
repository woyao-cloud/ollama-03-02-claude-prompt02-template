# 业务分析实施计划：创建模块化需求文档

## 背景
产品经理已完成产品需求文档（PRD.md）、用户画像（USER_PERSONAS.md）和产品路线图（ROADMAP.md）。根据项目CLAUDE.md规范，需要业务分析员将这些产品需求转化为六个功能模块的技术需求文档，以支持后续并行开发。

## 当前状态
- ✅ 产品经理输出文件完整：`prompts/product/PRD.md`、`USER_PERSONAS.md`、`ROADMAP.md`
- ❌ 模块化需求文档缺失：用户管理、角色权限管理、认证授权、审计日志、系统配置、部门管理
- ❌ 引用文档缺失：架构文档引用的FRD、NFRD、CONTEXT.md文件不存在
- 📋 现有PRD内容全面：包含12个用户故事、功能需求、非功能需求、部门管理详细需求

## 目标
创建标准化的模块化需求文档体系，将PRD内容拆分为六个独立模块，补充技术细节，确保文档支持后续架构设计、开发实施和测试验证。

## 六个功能模块
1. **用户管理** (`USER_MANAGEMENT.md`) - 用户注册、登录、资料管理、状态管理
2. **角色权限管理** (`ROLE_PERMISSION_MANAGEMENT.md`) - 角色、权限、分配、验证
3. **认证授权** (`AUTHENTICATION_AUTHORIZATION.md`) - JWT、OAuth2、会话管理、安全配置
4. **审计日志** (`AUDIT_LOG.md`) - 操作日志、登录日志、查询导出、保留策略
5. **系统配置** (`SYSTEM_CONFIGURATION.md`) - 系统参数、邮件模板、国际化、通知配置
6. **部门管理** (`DEPARTMENT_MANAGEMENT.md`) - 部门CRUD、树形结构、成员管理、权限继承

## 实施步骤

### 阶段1：建立文档结构（第1天）
1. **创建目录结构**
   ```
   prompts/requirements/
   ├── CONTEXT.md                    # 用户决策与技术约束文档 (v1.0)
   ├── FUNCTIONAL_REQUIREMENTS.md    # 功能需求文档 (FRD v1.2)
   ├── NON_FUNCTIONAL_REQUIREMENTS.md # 非功能需求文档 (NFRD v1.0)
   ├── USER_STORIES.md               # 整合的用户故事文档
   ├── USER_MANAGEMENT.md            # 用户管理需求
   ├── ROLE_PERMISSION_MANAGEMENT.md # 角色权限管理需求
   ├── AUTHENTICATION_AUTHORIZATION.md # 认证授权需求
   ├── AUDIT_LOG.md                  # 审计日志需求
   ├── SYSTEM_CONFIGURATION.md       # 系统配置需求
   └── DEPARTMENT_MANAGEMENT.md      # 部门管理需求
   ```

2. **创建标准化模板**
   - 制定 `REQUIREMENTS_TEMPLATE.md` 作为所有需求文档的基础模板
   - 模板包含：文档信息、概述、用户故事、功能需求、非功能需求、技术设计、API接口、数据模型、测试策略、部署要求

### 阶段2：分析产品需求（第1-2天）
1. **提取PRD核心内容**
   - 分析 `prompts/product/PRD.md`，提取12个用户故事（US-001至US-012）
   - 识别功能模块划分（5.1节：用户管理、部门管理、角色管理、权限管理、审计日志、安全模块、系统配置）
   - 提取非功能需求（6.1-6.5节：性能、安全、可用性、扩展性）
   - 特别关注部门管理详细需求（5.3节）

2. **分析用户画像**
   - 理解三种用户角色：系统管理员、部门经理、普通员工
   - 分析用户目标、痛点、使用场景
   - 映射用户角色到功能权限

3. **分析产品路线图**
   - 理解MVP功能范围（7.1节）
   - 规划迭代开发阶段
   - 确定各模块的优先级

### 阶段3：创建模块需求文档（第2-3天）
为每个模块创建详细需求文档，基于PRD内容补充技术细节：

#### 3.1 用户管理模块 (`USER_MANAGEMENT.md`)
- **用户故事**：US-001（创建用户）、US-002（用户自助注册）、US-003（用户资料管理）
- **核心功能**：用户CRUD、批量导入/导出、用户状态管理、密码重置
- **技术细节**：User实体设计、密码加密（BCrypt）、邮箱验证、用户状态枚举
- **API设计**：注册、登录、资料管理、密码重置接口

#### 3.2 角色权限管理模块 (`ROLE_PERMISSION_MANAGEMENT.md`)
- **用户故事**：US-006（创建角色）、US-007（分配用户角色）、US-008（细粒度权限控制）
- **核心功能**：角色CRUD、权限分配、角色继承、权限验证
- **技术细节**：Role和Permission实体、权限粒度（菜单/操作/字段/数据）、权限缓存（Redis）
- **API设计**：角色管理、权限分配、权限验证接口

#### 3.3 认证授权模块 (`AUTHENTICATION_AUTHORIZATION.md`)
- **用户故事**：US-011（密码策略管理）、US-012（双因素认证）
- **核心功能**：JWT认证、OAuth2集成、会话管理、安全配置
- **技术细节**：Spring Security配置、JWT令牌生成验证、双因素认证（TOTP/短信/邮箱）
- **API设计**：登录、登出、令牌刷新、OAuth2回调接口

#### 3.4 审计日志模块 (`AUDIT_LOG.md`)
- **用户故事**：US-009（查看操作日志）、US-010（个人操作历史）
- **核心功能**：操作日志记录、登录日志、日志查询筛选、日志导出
- **技术细节**：AuditLog实体、AOP日志切面、异步日志处理（Kafka）、日志保留策略
- **API设计**：日志查询、导出、统计分析接口

#### 3.5 系统配置模块 (`SYSTEM_CONFIGURATION.md`)
- **用户故事**：（从PRD功能需求中提取）
- **核心功能**：系统参数配置、邮件模板管理、通知设置、国际化配置
- **技术细节**：Configuration实体、配置热更新、配置缓存机制
- **API设计**：配置查询、更新、导入导出接口

#### 3.6 部门管理模块 (`DEPARTMENT_MANAGEMENT.md`)
- **用户故事**：US-004（管理部门树形结构）、US-005（管理部门成员）
- **核心功能**：部门CRUD、树形结构管理、部门成员管理、部门权限控制
- **技术细节**：Department实体、树形结构存储（Materialized Path）、部门权限继承、数据范围控制
- **API设计**：部门管理、成员管理、树形结构查询接口

### 阶段4：补充技术细节和整合文档（第3天）
1. **创建支持文档**
   - `CONTEXT.md`：用户决策与技术约束（引用现有架构文档内容）
   - `FUNCTIONAL_REQUIREMENTS.md` (FRD v1.2)：功能需求总览，汇总六个模块的核心功能，引用各模块文档
   - `NON_FUNCTIONAL_REQUIREMENTS.md` (NFRD v1.0)：整合PRD中的非功能需求，补充技术细节
   - `USER_STORIES.md`：整合所有用户故事，按模块分类

2. **补充技术设计**
   - 为每个模块定义详细的API接口规范（RESTful端点、请求/响应格式）
   - 设计数据模型（实体类定义、数据库表结构、关系图）
   - 定义业务逻辑关键算法

3. **验证一致性**
   - 检查各模块需求无冲突
   - 确保技术方案符合项目技术栈（Spring Boot 3.5 + Next.js 16+）
   - 验证非功能需求可测量（性能指标、安全要求）

### 阶段5：质量检查和交付（第3天）
1. **文档质量检查**
   - 所有文档使用统一模板
   - 技术细节完整准确
   - API接口定义清晰
   - 数据模型设计合理

2. **可实施性验证**
   - 需求可被分解为开发任务
   - 技术方案可行
   - 依赖关系清晰
   - 测试策略明确

## 关键决策点

1. **模块划分调整**：PRD中的"安全模块"将合并到"认证授权模块"中，包含密码策略、双因素认证、会话管理等。
2. **部门管理重点**：PRD中部门管理有详细需求（5.3节），需特别注意树形结构、权限继承、性能要求。
3. **技术细节粒度**：需求文档需包含足够技术细节（实体设计、API定义），但不过度设计。
4. **与非功能需求整合**：PRD中的非功能需求（性能、安全、可用性）需分配到各模块文档中。

- **风险**：PRD内容理解偏差
  - **缓解**：标记假设，创建问题清单，必要时与产品经理确认
- **风险**：技术细节不准确
  - **缓解**：参考现有架构文档，保持与技术架构一致
- **风险**：时间不足
  - **缓解**：优先完成核心模块（用户管理、认证授权、角色权限管理）
